# Pieter Levels — On Frontend Engineering

## Vanilla HTML, CSS, and JavaScript

Levels' frontend engineering is deliberately minimal: HTML, CSS, and enough JavaScript to make forms submit and toggle visibility. No React, no Vue, no build pipeline, no bundler. The production CSS is written by hand. The JavaScript is inline or in a single script file.

This is not a limitation — it is a deliberate engineering choice. A frontend with no build pipeline deploys instantly (copy files to the server). A frontend with no framework has no framework to update when security vulnerabilities are discovered. A frontend in plain HTML is readable by any developer, debuggable in any browser's DevTools, and understandable to the developer who wrote it six months later.

## Performance by Absence

Levels' frontend is fast not because it is optimised but because there is nothing to slow it down. No JavaScript bundle to parse and execute. No React reconciliation pass on every render. No font loading from a third-party CDN. The HTML arrives from the server and renders immediately.

The frontend engineering principle: the fastest frontend is the one with the fewest moving parts. Before adding a library, framework, or asset, ask whether the page is measurably faster without it. If the page is already fast, the addition is a regression risk.

## PHP Templates as Frontend Architecture

Levels' frontend HTML is generated by PHP templates — server-side rendering with no client-side component framework. The page is HTML when it arrives at the browser; JavaScript enhances it rather than generating it. This is the architecture Hansson advocates with Hotwire, implemented with PHP instead of Rails.

The engineering argument: server-rendered HTML is faster to display than client-rendered HTML because the browser can start rendering immediately upon receiving the first bytes, rather than waiting for a JavaScript bundle to download, parse, and execute.

## CSS Without Frameworks

Levels does not use Bootstrap, Tailwind, or CSS-in-JS. His CSS is written by hand — class names describe the element or its function, not its visual appearance. The engineering benefit: the CSS is a direct expression of the visual design, without the translation layer of a utility framework. When the design changes, one CSS rule changes.

The frontend engineering discipline for solo developers: write CSS you can read and maintain. A utility class framework reduces initial CSS authoring time but adds a translation layer between the visual design and the code. For small teams, this trade-off often does not favour the framework.

## Progressive Enhancement as Default

Levels' frontend works without JavaScript. Forms submit to the server and the page reloads. Links navigate to new pages. This is not a performance optimisation — it is the architectural default. JavaScript is added only where interaction genuinely requires it.

The engineering benefit: a page that works without JavaScript works in every browser, works when JavaScript is blocked by a corporate proxy, and works for users who have disabled JavaScript for performance or privacy reasons. It is also easier to debug: the server is the source of truth, not a combination of server state and client state.
