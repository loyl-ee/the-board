# Chris Eidhof — Quotes

All quotes below are verbatim from primary sources. Sources are linked where possible.

---

## On Functional Programming

> "What I love about Swift is that it allows us to mix functional and object-oriented programming."
— Chris Eidhof, "Functional Programming in Swift" (chris.eidhof.nl/post/functional-programming-in-swift/), 2014

> "The book is not about any specific frameworks, it's about doing functional programming. Frameworks might change a lot, the principles of functional programming have been the same for a really long time."
— Chris Eidhof, "Functional Programming in Swift" (chris.eidhof.nl/post/functional-programming-in-swift/), 2014

> "Functional programming can be like that: making certain programs smaller by a huge factor."
— Chris Eidhof, "Functional Programming in Swift" (chris.eidhof.nl/post/functional-programming-in-swift/), 2014

---

## On Types and the Compiler

> "A type checker actually does testing for you. It's not a replacement for TDD, but it allows you to completely get rid of a whole bunch of tests."
— Chris Eidhof, "Types vs TDD" (chris.eidhof.nl/post/types-vs-tdd/)

> "I think of the compile errors as a todo-list, not as a speed bump."
— Chris Eidhof, "Types vs TDD" (chris.eidhof.nl/post/types-vs-tdd/)

> "Types are not a silver bullet. You still need to test your code. But wouldn't you rather test interesting parts, and leave the boring stuff to the compiler?"
— Chris Eidhof, "Types vs TDD" (chris.eidhof.nl/post/types-vs-tdd/)

## On Value Semantics

> "if you read a line like `let point = ...`, and you know that `point` is a struct variable, then you also know that it will never, ever, change."
— Chris Eidhof, "Structs and Mutation in Swift" (chris.eidhof.nl/post/structs-and-mutation-in-swift/), 2016

> "Because every struct variable is unique (or in other words: every struct value has exactly one owner), it's almost impossible to introduce bugs this way."
— Chris Eidhof, "Structs and Mutation in Swift" (chris.eidhof.nl/post/structs-and-mutation-in-swift/), 2016

> "Because they are defined as structs, they have copy semantics. If you have a constant string, there is no way it can change under your nose."
— Chris Eidhof, "Struct Semantics in Swift" (chris.eidhof.nl/post/struct-semantics-in-swift/), 2015

## On API Design and Simplicity

> "I'm a big fan of using the simplest possible solution for your problem and team."
— Chris Eidhof, "Swift Analytics" (chris.eidhof.nl/post/swift-analytics/)

> "I think all solutions are valid. They're different embeddings of the same problem."
— Chris Eidhof, "Swift Analytics" (chris.eidhof.nl/post/swift-analytics/)

## On View Controllers and Architecture

> "Making view controllers simpler has been a long obsession for me."
— Chris Eidhof, "Functional View Controllers" (chris.eidhof.nl/post/functional-view-controllers/)

> "In functional view controllers, every view controller takes some input, renders that, and has some output."
— Chris Eidhof, "Functional View Controllers" (chris.eidhof.nl/post/functional-view-controllers/)

## On SwiftUI and Teaching

> "It's what I wished SwiftUI documentation would be like: way more visual and interactive."
— Chris Eidhof, "2024 in Review" (chris.eidhof.nl/post/2024-review/), 2024

> "The learning has been tremendous from all sides."
— Chris Eidhof, "2024 in Review" (chris.eidhof.nl/post/2024-review/), 2024 (on teaching SwiftUI workshops)

> "View trees are ephemeral, and render trees are persistent. That's very important to keep in mind."
— Chris Eidhof, "A Day in the Life of a SwiftUI View" (chris.eidhof.nl/presentations/day-in-the-life/)

— Chris Eidhof, "Functional Programming in Swift" (chris.eidhof.nl/post/functional-programming-in-swift/), 2014 (on the decision to write *Functional Swift*)
